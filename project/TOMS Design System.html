TOMS Design System

Direction A — Refined Enterprise. A clear, calm, information-dense system anchored in TOMS blue. Designed for operators who spend hours a day inside the platform managing fleets, pushing apps, and resolving incidents.

Brand

The mark pairs a cloud crown (infrastructure) with an infinity loop (always-on device fleet). Gradient runs cyan → blue → violet to signal real-time telemetry.

Device status — a two-dimensional model

Every terminal has two independent status dimensions. Connectivity answers "can we talk to it?"; inventory answers "what is its lifecycle state?". Display them together — never collapse into one field.

Connectivity

Online Offline

Pulses while reporting. Derived from last-seen heartbeat within 90 s.

Inventory

In Use In Stock Transferring Locked Repair Lost Retired

Operator-managed lifecycle. Changes via Transfer, Lock, Decommission actions.

Command flow

Every remote command goes through the same three states: Confirm → Running → Done. Destructive commands add an extra typed-confirm step.

action
  ↓  confirm modal (secondary · primary/danger)
  ↓  running (indeterminate progress · 30s ack window)
  ↓  done (success toast · device timestamp)

Do & don't

✓ Do
Show connectivity and inventory as independent signals. Use the dot for connectivity, the pill for inventory.
✗ Don't
Collapse both into one status column — "Offline / Repair" is two facts, not one.
✓ Do
Reserve danger red for destructive actions. Use warn amber for recoverable issues.
✗ Don't
Style routine warnings in red — operators will numb to real alerts.
✓ Do
Keep monospace for identifiers (SN, MAC, IP, firmware). It aids scanning.
✗ Don't
Mix serial numbers into proportional text — they become hard to compare.
